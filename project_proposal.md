## 3. 프로젝트 추진 배경 및 필요성

### 가. 기술적 배경
*   **클라우드 환경의 동적 자원 할당 및 Auto Scaling**: 현대의 클라우드 컴퓨팅 환경에서는 트래픽과 연산 부하의 변화에 맞춰 서버 자원을 동적으로 늘리거나 줄이는 Auto Scaling과 자원 할당 최적화 기술이 시스템의 가용성과 비용 효율을 결정하는 핵심 요소입니다.
*   **하부 인프라의 자원 격리(cGroup) 및 분산 처리 제어**: 이러한 클라우드의 동적 자원 관리가 가능하려면, 하부 시스템 레이어에서 리눅스 커널의 cGroup 등을 이용한 정밀한 자원 격리가 선행되어야 합니다. 또한, 할당된 이기종 자원들을 하나로 묶어 효율적으로 연산을 분배하는 분산 런타임 프레임워크(Ray 등)의 고도화가 필수적인 기술적 배경으로 대두되고 있습니다.

### 나. 요구사항 분석
*   **클라우드 중심에서 코어 메커니즘 중심(로컬 Docker)으로의 접근 방식 피벗**: 본 과제는 당초 퍼블릭 클라우드 환경(오픈스택, AWS 등) 상에서 가상 자원 할당 및 Auto Scaling을 직접 구현하고 벤치마크하는 것을 목표로 하였습니다. 그러나 단순히 방대한 클라우드 환경을 세팅하는 것에 집착하기보다는, **클라우드 인프라의 가장 깊은 뿌리인 '컨테이너 자원 격리(cGroup)'와 '분산 작업 스케줄링'의 작동 원리를 파악**하는 것이 엔지니어링적으로 더 가치 있다는 요구사항 분석에 도달했습니다.
*   **가상 클러스터 모사 및 실제 워크로드 기반 실증**: 이에 따라 고비용의 클라우드를 띄우는 대신, 로컬 단일 호스트 내에서 Docker Compose를 이용해 클라우드 서버 환경을 모사하고, 실제 머신러닝 연산 부하(PyTorch LSTM)를 가해 자원 격리 상태에서의 시스템 성능을 실증하는 구조가 요구됩니다.

### 다. 수행 필요성
*   **클라우드 자원 관리의 본질적 이해 및 분산 엔진 개발 역량 함양**: 로컬 Docker 환경에서 경량 분산 런타임(Baby Ray)을 바닥부터 구축해봄으로써, 상용 클라우드가 하부에서 실제로 가상머신/컨테이너의 자원을 어떻게 제어하고(cGroup), 병목 발생 시 어떻게 다른 노드로 작업을 우회시키며(동적 스케줄링), 장애 발생 시 어떻게 복원해내는지(Auto Healing/Reconstruction)를 소스 코드 레벨에서 명확히 실증할 수 있습니다. 
*   단순히 상용 클라우드의 겉면 API를 조작하는 것을 넘어, 인프라의 근간을 직접 설계하고 트러블슈팅하는 심도 있는 클라우드 엔지니어링 역량을 증명하기 위해 본 수행 방향이 반드시 필요합니다.

## 4. 최종 목표 및 주요 결과물

### 가. 최종 목표
*   **경량 분산 런타임 플랫폼 구축**: Docker Compose를 활용하여 1개의 Head Node와 2개의 Worker Node로 이루어진 가상의 다중 분산 클러스터를 가동하고 gRPC 통신 레이어를 구현함.
*   **PyTorch LSTM 학습 태스크 분산 스케줄링 실증**: 시계열 데이터를 학습하는 다수의 LSTM 태스크를 각 노드별 자원(cGroup 격리 한계치)에 맞추어 지연 시간을 최소화하는 스마트 스케줄러를 개발함.
*   **결함 허용(Fault Tolerance) 안전장치 완성**: | **중점 개선내용** | **[기존방식 (초기 기획의 한계)]**<br>• 당초 퍼블릭 클라우드 상에서 가상 자원 할당 및 Auto Scaling 환경을 구축하려 했으나, 방대한 클라우드 세팅에 매몰되어 정작 인프라 하부의 코어 기술(자원 격리 원리, 스케줄링 로직)을 딥다이브하여 연구하기에는 한계가 존재함.<br><br>**[개선방향 (해결방안)]**<br>1. **클라우드 코어 환경의 로컬 모사 (피봇팅)**: 거대 클라우드 환경에 집착하지 않고, 로컬 물리 PC 내에서 Docker Compose와 리눅스 cGroup을 이용해 클라우드 서버의 자원 제어 환경을 완벽히 모사하는 방향으로 선회함.<br>2. **gRPC 기반 분산 통신망 직접 구현**: 클라우드 내 가상 서버 간의 생존 신호(Heartbeat) 파악 및 작업 지시를 위한 통신망을 Protobuf와 gRPC를 이용해 바닥부터 직접 구축함.<br>3. **cGroup 격리 및 PyTorch LSTM 부하 연동**: 작업 노드별로 CPU/Memory 한도를 다르게 한정(cGroup 격리)하고, 인위적인 지연 대신 실제 PyTorch LSTM 학습 코드를 돌려 물리적 자원 제한에 따른 진짜 성능 비대칭 환경을 조성함.<br>4. **자원 인지형 동적 스케줄링 (Auto Scaling의 근간)**: 각 노드의 남은 cGroup 자원과 대기 큐 깊이를 확인해 성능이 넉넉한 노드로 작업을 집중 배치하거나 우회(버스팅)시키는 오프로딩 코드를 작성하여 동적 부하 분산의 원리를 증명함.<br>5. **Task Lineage 기반 장애 복구 (Auto Healing)**: 작업 노드가 강제 중단되면 대장 서버에 기록된 족보(DAG)를 거슬러 올라가 누락된 부분만 재계산해 복원하는 예외 대응 로직을 완성함. |

| **중점 개선내용** | **[기존방식 (또는 문제점)]**<br>• 이미 구축된 분산 프레임워크의 상위 API만 사용하면 하부 시스템 레이어(네트워크 전송 오버헤드, 자원 격리 시의 노드 병목, 크래시 발생 시의 내부 재연산 흐름)를 직접 튜닝하고 구조를 파악하기 어려움.<br><br>**[개선방향 (해결방안)]**<br>1. **gRPC 기반 분산 통신망 구현**: 메인 제어기(Head)와 작업 서버(Worker) 간 전송 메시지 규격(Protobuf)을 정하고 연결성(Heartbeat) 파이프라인을 직접 구축함.<br>2. **cGroup 격리 및 PyTorch LSTM 부하 연동**: 작업 노드별로 CPU/Memory 한도를 다르게 한정(cGroup 격리)하고, 인위적인 sleep 지연 대신 실제 PyTorch LSTM 학습 코드를 돌려 물리적 자원에 따른 진짜 성능 비대칭 환경을 실증함.<br>3. **자원 인지형 동적 스케줄링**: 각 노드의 남은 cGroup 자원과 대기 큐 깊이를 확인해 성능이 넉넉한 1번 노드로 작업을 집중 배치하거나 붐비는 작업을 우회(버스팅)하는 오프로딩 코드를 작성함.<br>4. **Task Lineage 기반 장애 복구**: 학습 데이터의 조리 순서(Task Lineage DAG)를 대장 서버에 기록해 두고, 작업 노드가 강제 중단되면 사라진 조합의 학습 작업만 다른 노드에서 재계산해 복원하는 예외 대응 로직을 완성함. |
| **설계 참고사항** | **[일꾼 노드 수 최소화 및 확장성 검증 기획]**<br>• **2대 노드 기반 기능 검증**: 가상 자원 제한, 지역성 스케줄링, 장애 시 리니지 복구 등의 분산 컴퓨팅 핵심 기능들은 Worker 노드 2대로 모두 구현 및 실증이 가능하므로 핵심 뼈대는 2대 구조로 구성하여 복잡성을 통제함.<br>• **Scale Out 실험 병행**: 스케줄러 코드가 가변 노드를 수용할 수 있게 설계하고, 최종 벤치마크 단계에서 `docker-compose scale` 옵션을 연동하여 Worker 수를 2대 ➔ 3대 ➔ 4대로 확장하면서 처리 성능 곡선을 정량적으로 측정함. |
| **결과물 활용** | • AI 분산 학습 인프라 구축이나 가상 서버 동적 부하 분산 로직 설계 시 참고할 수 있는 기준 아키텍처 모델로 활용함.<br>• 분산망 메시지 연동 및 장애 시 자동 복구(Reconstruction) 동작 원리를 테스트할 수 있는 오픈소스 교육용 경량 분산 시뮬레이터 패키지로 릴리즈함. |

#### ▢ 수행방법
*   ❍ **gRPC 기반 가상 분산 노드 및 통신 프로토콜 설계**
    *   Docker Compose로 1개의 Head 노드와 2개의 Worker 노드를 띄우고 내부 사설 가상망을 구성합니다.
    *   노드 간 자원 점유 상태 보고(Heartbeat) 및 태스크 작업 할당 메시지 형식(Protobuf)을 직접 선언하여 gRPC 통신 서비스를 구현합니다.
*   ❍ **cGroup 자원 격리 및 PyTorch LSTM 학습 태스크 연동**
    *   각 작업 노드의 CPU/메모리 한계를 다르게 격리(Worker 1: 2.0 Cores / Memory 2GB, Worker 2: 0.5 Cores / Memory 512MB)합니다.
    *   노드별 cGroup CPU 설정에 맞춰 `torch.set_num_threads`를 명시적으로 연계하고, 하이퍼파라미터 조합별 LSTM 시계열 학습 태스크를 독립적으로 구동시켜 실제 물리 자원에 비례한 학습 속도 차이를 실증합니다.
*   ❍ **데이터 지역성(Locality) 인지형 작업 배치 및 동적 버스팅 구현**
    *   대용량 학습 데이터를 전송하는 직렬화 지연 시간을 분석하고, 학습 데이터가 이미 로드된 컨테이너 노드에 해당 연산을 우선 할당하여 데이터 전송 오버헤드를 줄이는 스케줄러를 개발합니다.
    *   특정 노드가 포화 상태에 이르면 작업을 자원 여유가 있는 노드로 동적으로 옮기는(버스팅) 오프로딩 스케줄링 엔진 코드를 통합합니다.
*   ❍ **Task Lineage 기반 자동 장애 복구 실증**
    *   연산 실행 도중 특정 컨테이너를 `docker stop`으로 강제 종료시킵니다.
    *   대장 서버가 하트비트 두절을 확인하면 GCS에 기록된 작업 이력서(Task Lineage DAG)를 역추적해 유실된 하이퍼파라미터 조합의 학습 태스크를 식별하고, 살아있는 노드에서 그 작업만 자동으로 재실행(Re-execution)하여 최종 loss 데이터를 유실 없이 정상 복구함을 실증합니다.
